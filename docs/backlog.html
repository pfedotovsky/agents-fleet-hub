Items

☐ 1. Answer the model's questions (AskUserQuestion) UXno new API

Now

When the agent asks a question via AskUserQuestion, it arrives as a plain permission_request and the hub shows only the generic Allow / Deny prompt — there is no way to actually pick an answer, so the agent gets blocked or answers itself.

Want
  • Detect toolName === 'AskUserQuestion' the same way isPlanRequest() special-cases ExitPlanMode.
  • Render a question card: question text, header chip, one button per option (plus multi-select support), and a free-text "Other" input.
  • Return the selected answers through the permission response (updatedInput / structured answer payload — verify the exact shape against a live host before building).
  • Multiple questions in one request → stepper or stacked groups.
Touches: ChatPane.tsx (new QuestionCard next to PlanApprovalCard), chatSocket.ts (respond payload), types.ts

☐ 2. Plan mode as a separate toggle (Shift+Tab) + plan side panel UX

Now

"Plan mode" is just one option inside the permissions <select> (PERMISSION_MODES), so it competes with Ask/Accept edits/Bypass and resets awkwardly after plan approval. The finished plan renders inline in the chat stream and scrolls away.

Want
  • Remove plan from the permission-mode select; make it an independent boolean toggle in the composer (visible pill/switch, e.g. "Plan").
  • Shift+Tab in the composer cycles the toggle (matches Claude Code CLI muscle memory); permission mode stays whatever it was.
  • Render the plan in a side panel (right-hand drawer, like Cursor Agents on Windows): sticky while the chat continues, with Approve / Approve+accept-edits / Revise actions in the panel header. The inline card shrinks to a one-line "Plan ready — open panel" chip.
  • Keep the existing behavior where approval drops plan mode locally (see comment at respondPlan), now expressed as toggling the switch off.
Touches: ChatPane.tsx (PERMISSION_MODES, PlanApprovalCard, composer keybinding), new PlanPanel.tsx, App.tsx (layout slot for the drawer)

☐ 3. Much wider message text — CloudCLI-style density UX

Now

Readable width is capped twice: the chat column is max-w-[54rem] (ChatPane.tsx) and assistant text is further capped at max-w-[46rem] (Messages.tsx), plus px-6 gutters. On a wide monitor most of the pane is empty; CloudCLI's web UI lets text use nearly the full pane.

Want
  • Drop the inner max-w-[46rem] caps on assistant messages / tool cards — let them fill the column.
  • Raise the column cap substantially (e.g. max-w-[80rem]) or remove it and keep only small gutters; shrink px-6 → px-3/px-4 like CloudCLI.
  • Apply the same width to the composer (max-w-[54rem] in the footer) so input and messages align.
  • Options to evaluate: a width preference (comfortable / wide / full), or width that scales with the pane via clamp(). Long-form prose readability is the counter-argument — test with real transcripts before settling.
  • Code blocks and diffs benefit most — they currently wrap/scroll inside the narrow cap.
Touches: ChatPane.tsx (lines ~941, ~982), Messages.tsx (lines 60, 83, 90), ToolCall.tsx, Diff.tsx
